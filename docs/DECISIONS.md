## 0. 용어부터 — 이게 다 뭔데?

표를 보기 전에. 비개발 회원도 이해할 수 있게.

**한 줄 비유:** Postgres = 데이터 넣는 창고 · Supabase = 그 창고를 빌려주고 관리해주는 서비스 · Next.js = 화면+서버를 한 번에 만드는 건축 도구.

### 핵심 3가지

| 용어 | 한마디 | 자세히 |
|------|--------|--------|
| **Postgres** (PostgreSQL) | 데이터베이스(똑똑한 공용 엑셀) | 데이터를 표(행/열)로 저장. 여러 명이 동시에 써도 안 꼬임. 회원·글·프로젝트를 여기 저장하고 `SQL`이란 언어로 꺼냄. 무료·오픈소스·업계 표준 → 인계 유리. |
| **Supabase** | Postgres 창고 임대 서비스 | Postgres를 직접 깔고 관리(설치·백업·보안)하는 건 귀찮음 → 클라우드에서 클릭 몇 번으로 띄워주고 대신 관리. 로그인·파일 기능도 끼워팔지만 **우리는 "DB 창고만" 빌려 씀**(나머지까지 쓰면 락인돼서 쿠러그 이전 불가). |
| **Next.js** | 화면+서버를 한 번에 만드는 도구 | 웹사이트는 보통 ①화면(프론트) ②서버(백엔드)가 따로 필요한데, Next.js는 **둘을 한 프로젝트에서** 만들게 해줌. 바탕은 React(화면을 레고처럼 조립). 그래서 "별도 백엔드 없이 Next 하나로 충분". |

### 어떻게 맞물리나

| 흐름 | 담당 |
|------|------|
| 회원 브라우저 — 화면 보기/글쓰기 | **Next.js** (화면+서버 다 담당) |
| ↳ 데이터(회원·글·프로젝트) | **Postgres** (Supabase가 빌려줌) |
| ↳ 파일(족보 PDF·이미지) | **R2** (파일 전용 창고) |
| ↳ 로그인 | **GitHub** (OAuth로 인증) |

### 나머지 용어

| 용어 | 설명 |
|------|------|
| **React** | 화면을 컴포넌트(레고)로 조립하는 라이브러리. Next.js의 바탕. |
| **ORM** (Prisma/Drizzle) | 코드 ↔ DB 번역기. SQL 직접 안 쓰고 코드로 DB 다룸 → 안전하고 DB 갈아끼우기 쉬움. |
| **OAuth** | "GitHub/구글 계정으로 로그인" 표준. 우리가 비번 안 받고 GitHub가 대신 인증. |
| **Auth.js** (NextAuth) | Next.js에 그 로그인 기능을 쉽게 붙여주는 라이브러리. |
| **S3 / R2 / MinIO** | 파일 저장 전용 창고. S3=아마존 원조(표준), R2=Cloudflare판(전송료 무료), MinIO=자기 서버에 까는 버전. **셋이 호환**이라 코드 그대로 갈아끼움. |
| **Vercel** | Next.js 만든 회사의 배포 서비스. GitHub에 올리면 자동으로 사이트 띄워줌(개발 단계용). |
| **Docker / 컨테이너** | 앱+실행환경을 통째로 박스에 포장 → 어느 서버든 박스만 열면 똑같이 돌아감. 쿠러그 이전이 쉬워지는 이유. |
| **락인 / 포터블** | 락인=특정 회사 서비스에 갇혀 못 옮김 / 포터블=어디든 옮기게 짜기. |

## 1. 아키텍처 `확정`

코드를 어떻게 담을지. 기여자가 매학기 바뀌므로 "세팅 단순함"이 최우선 기준.

| 옵션 | 스택/방식 | 장점 | 단점 |
|------|-----------|------|------|
| **단일 통합 ✅** | Next.js 하나(App Router + API Routes) | 세팅 1번, 진입 쉬움, 배포 단순 | 백엔드 아주 커지면 분리 필요할 수도 |
| 모노레포 | Turborepo에 web/api 폴더 | 추후 분리 여지 | 초기 복잡도↑, 학생팀엔 과함 |
| 완전 분리 🔴 | Next + Spring/Nest 별도 | 백엔드 학습폭↑ | 세팅·배포·유지 2배, 무산 리스크↑ |

👉 **추천: Next.js 단일 통합.** 분리는 진짜 필요해질 때 해도 늦지 않음. (옛 레포의 React+Spring 분리안은 이 결정으로 폐기)

## 3. DB `확정`

데이터 저장. 쿠러그 이전 위해 **포터블**해야 함.

| 옵션 | 스택 | 장점 | 단점 |
|------|------|------|------|
| **Postgres(Supabase로 시작) ✅** | 매니지드 Postgres + ORM | 무료 시작, SQL표준, 순수 Postgres라 포터블 | 무료는 7일 무접속 시 일시정지(Cron 핑으로 회피) |
| Neon | 서버리스 Postgres | 무료 넉넉, 일시정지 없음, 브랜칭 | 비교적 신생 |
| Supabase Auth/RLS까지 사용 🔴 | Supabase 풀기능 | 인증·실시간 공짜로 따라옴 | 쿠러그 이전 시 락인 |
| 자가호스팅 Postgres | 쿠러그 서버 컨테이너 | 비용0, 최종 종착지 | 학기 초까지 대기 |

👉 **추천: 순수 Postgres + ORM.** Supabase는 "DB만" 빌려 쓰고 Auth/RLS는 안 씀 → 이전 시 커넥션 문자열만 교체.

## 4. 파일 저장 (족보) `확정`

족보 PDF·이미지. 용량과 "이전 쉬움"이 관건.

| 옵션 | 스택 | 장점 | 단점 |
|------|------|------|------|
| **Cloudflare R2 (S3 SDK로) ✅** | `@aws-sdk/client-s3` | 10GB 무료·전송료 0, MinIO로 이전 쉬움 | 계정 1개 추가 |
| Supabase Storage 🔴 | Supabase | 한 곳에서 관리 | 무료 1GB뿐(족보로 금방 참), 락인 |
| AWS S3 🔴 | S3 | 표준·무제한 | 전송료 과금, 결제·IAM 복잡 |
| MinIO(쿠러그 서버) | 자가호스팅 S3호환 | 비용0, 종착지 | 학기 초 대기 |

👉 **추천: R2를 S3 호환 SDK로.** 코드를 S3 규격으로 짜두면 R2(지금)→쿠러그 MinIO(나중)로 env만 바꿔 이전.
