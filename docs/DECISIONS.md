# T.G.winG 웹 — 기술·운영 결정

호스팅 종착지는 쿠러그(KU LUG) 자체 서버. 그래서 클라우드 락인을 피하는 "포터블" 제약이 곳곳에 붙는다.
✅ 채택 · 🔴 비추천.

---

## 1. 아키텍처 `확정`

기여자가 매학기 바뀌므로 세팅 단순함이 최우선.

| 옵션 | 스택/방식 | 장점 | 단점 |
|------|-----------|------|------|
| **단일 통합 ✅** | Next.js 하나(App Router + API Routes) | 세팅 1번, 진입 쉬움, 배포 단순 | 백엔드 아주 커지면 분리 필요할 수도 |
| 모노레포 | Turborepo에 web/api 폴더 | 추후 분리 여지 | 초기 복잡도↑ |
| 완전 분리 🔴 | Next + Spring/Nest 별도 | 백엔드 학습폭↑ | 세팅·배포·유지 2배, 무산 리스크↑ |

👉 **Next.js 단일 통합.** 분리는 진짜 필요해질 때. (옛 레포 React+Spring 분리안 폐기)

| 옵션 | 스택 | 장점 | 단점 |

👉 **Next.js.** 자료량·생태계·이력서 가치.

## 3. DB `확정`

쿠러그 이전 위해 포터블해야 함.

| 옵션 | 스택 | 장점 | 단점 |
|------|------|------|------|
| **Postgres(Supabase로 시작) ✅** | 매니지드 Postgres + ORM | 무료 시작, SQL표준, 순수 Postgres라 포터블 | 무료는 7일 무접속 시 일시정지(Cron 핑으로 회피) |
| Neon | 서버리스 Postgres | 무료 넉넉, 일시정지 없음, 브랜칭 | 비교적 신생 |
| Supabase Auth/RLS까지 사용 🔴 | Supabase 풀기능 | 인증·실시간 따라옴 | 이전 시 락인 |
| 자가호스팅 Postgres | 쿠러그 서버 컨테이너 | 비용0, 종착지 | 학기 초까지 대기 |

👉 **순수 Postgres + ORM(Drizzle/Prisma).** Supabase는 DB만, Auth/RLS는 안 씀 → 이전 시 커넥션 문자열만 교체.

## 4. 파일 저장 (족보) `확정`

| 옵션 | 스택 | 장점 | 단점 |
|------|------|------|------|
| **Cloudflare R2 (S3 SDK) ✅** | `@aws-sdk/client-s3` | 10GB 무료·전송료 0, MinIO로 이전 쉬움 | 계정 1개 추가 |
| Supabase Storage 🔴 | Supabase | 한 곳 관리 | 무료 1GB뿐, 락인 |
| AWS S3 🔴 | S3 | 표준·무제한 | 전송료 과금, IAM 복잡 |
| MinIO(쿠러그 서버) | 자가호스팅 S3호환 | 비용0, 종착지 | 학기 초 대기 |

👉 **R2 + S3 호환 SDK.** S3 규격으로 짜두면 R2 → 쿠러그 MinIO를 env 교체로 이전.

## 5. 인증 (로그인 + 멤버 판정) `확정`

| 옵션 | 스택 | 장점 | 단점 |
|------|------|------|------|
| **Auth.js + GitHub OAuth + Org 체크 ✅** | NextAuth | 무료·포터블, Org 멤버=자동 멤버 | Org 초대 관리 필요 |
| Supabase Auth 🔴 | Supabase | 세팅 쉬움 | 락인 |
| Clerk | SaaS | UI 완성도 | 무료한도 후 유료, 외부 의존 |
| Lucia | 자체 구현 | 완전 제어·포터블 | 직접 짤 게 많음 |
| 학번·이름 DB 대조 🔴 | 자체 | 회원가입 없이 간단 | 학번·이름은 공개정보 → 사칭 가능(인증 무의미) |

👉 **Auth.js + GitHub OAuth, 멤버 판정 = GitHub Org 멤버십.** 승인 테이블 없이 Org 초대로 끝. (옛 레포 "학번·이름 대조"는 보안상 폐기)
