## 1. 아키텍처 `확정`

기여자가 매학기 바뀌므로 세팅 단순함이 최우선.

| 옵션 | 스택/방식 | 장점 | 단점 |
|------|-----------|------|------|
| **단일 통합 ✅** | Next.js 하나(App Router + API Routes) | 세팅 1번, 진입 쉬움, 배포 단순 | 백엔드 아주 커지면 분리 필요할 수도 |
| 모노레포 | Turborepo에 web/api 폴더 | 추후 분리 여지 | 초기 복잡도↑ |
| 완전 분리 🔴 | Next + Spring/Nest 별도 | 백엔드 학습폭↑ | 세팅·배포·유지 2배, 무산 리스크↑ |

👉 **Next.js 단일 통합.** 분리는 진짜 필요해질 때. (옛 레포 React+Spring 분리안 폐기)

> 회의 합의(2026-06-26): 현재 기능 명세(족보 저장·커뮤니티)는 단일레포로 충분한 규모. 분리하면 옛 레포처럼 무산 위험↑. 풀스택이라 프론트도 백엔드 지식이 어느 정도 필요하지만(이슈 원인·수정을 AI에게 설명할 정도), 규모가 작아 일부러 역할을 가르지 않고 필요할 때 공부하며 진행하기로.

### 단일레포 ≠ 풀스택 강제 — 폴더 경계로 역할 분리

프론트만 하는 사람도 백엔드 안 건드리고 작업 가능. App Router가 원래 이 분리를 전제로 설계됨.

```
src/
├─ app/
│  ├─ (public)/          # 페이지 (프론트 주력)
│  └─ api/               # route handlers (백 담당 구역)
├─ components/           # 프론트 전용 (순수 React, 서버코드 import 금지)
│  ├─ ui/                # 디자인 시스템
│  └─ features/          # 도메인 UI
├─ lib/                  # 백 전용 (DB·인증·R2)
│  ├─ db/                # Drizzle 스키마·쿼리
│  ├─ auth.ts            # Auth.js 설정
│  └─ storage.ts         # R2 클라이언트
└─ types/                # 공유 계약 (프론트·백 둘 다 import)
```

- **담당 구역:** 프론트 = `app/` + `components/`, 백 = `lib/` + `app/api/`. 풀스택 한 명이 다 할 필요 없음.
- **연결 방식:** 백이 `lib/`에 함수 시그니처(빈 구현)만 먼저 박고 `types/`에 계약 정의 → 프론트는 그 타입만 보고 더미 데이터로 UI를 끝까지 완성. 내부 구현(DB 쿼리)은 몰라도 됨.
- **안전장치:** ESLint `import/no-restricted-paths`로 `components/` → `lib/db`·`lib/auth` import 차단 + 서버 모듈에 `import 'server-only'` → 클라이언트 번들에 새면 빌드 에러.

> **리뷰 반박 반영(2026-06-28, 권구현):** "GitHub만으로 회원관리는 위험" 지적은 우리 상황엔 과한 우려 — **컴공/개발 동아리**라 회원이 GitHub 계정을 갖는 게 자연스럽고, 비개발 회원 배제는 의도된 선이다. 별도 members 테이블·관리 사이트는 안 만든다. 단 두 가지만 반영: ① 멤버십은 **로그인 시점에 1회 확인해 세션에 굽는다**(Auth.js `signIn`/`session` 콜백) → GitHub Org API가 매 요청 실시간 의존이 되지 않음(장애·rate limit가 로그인 장애로 번지는 것 차단). ② "active 회원" = 현재 Org 멤버. 동비 안 낸 사람·졸업·탈퇴자는 운영진이 **매학기 Org에서 정리**(권한 잔존 방지).

## 14. 브랜치 전략 & 보호 `확정`

## 15. 가용성 / failover `확정`

쿠러그 다운 시 대응. 핵심 함정: 앱은 어디든 띄우지만 **DB·파일이 쿠러그 안에 있으면 넘겨도 빈 껍데기.**

| 옵션 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **(A) 수동 failover ✅** | 포터블 + Vercel 웜 스탠바이, 다운 시 DNS 플립 | 거의 공짜, 단순 | 복구에 몇 분, 수동 |
| (B) 즉시 failover | compute만 쿠러그, DB(Supabase/Neon)+파일(R2)은 매니지드 유지 | DNS 플립으로 즉시, 백업 외부 보관 | 완전 자가호스팅 포기, 소액 비용·외부 의존 |
| (C) 완전 자가호스팅 | DB·파일까지 쿠러그 | 비용0·독립 최고 | failover 사실상 불가 |
| 자동 failover 🔴 | health-check + DNS LB + 데이터 복제 | 무중단 | 유지보수 지옥=무산 원인 |

👉 **(A) 수동 failover 기본.** 가용성이 더 중요하면 (B). 자동 failover는 만들지 않는다. 쿠러그는 전원복구 자동부팅+원격리부팅+전담운영이라 다운 자체가 드묾.

스캐폴딩 시 반드시 만족(= 스캐폴딩 "완료"의 정의):

| 항목 | 방식 | 효과 |
|------|------|------|
| **원커맨드 부팅** | `cp .env.example .env && docker compose up -d db && npm run dev` (또는 `npm run setup` 한 방) | 실키 없이 바로 실행 |
| **로컬 DB** | docker-compose Postgres + `npm run seed`(가짜 프로젝트·블로그 데이터) | Supabase 계정 불필요, 빈 화면 방지 |
| **인증 우회** | `AUTH_DEV_BYPASS=true`(.env.example 기본값) → 가짜 멤버 세션 | GitHub OAuth 앱·시크릿 불필요 |
| **저장소 우회** | dev에선 업로드를 로컬 `./uploads`(또는 public/)에 기록 | R2 키 불필요 |
| **.env.example가 곧 동작** | dev 기본값 다 채워둠, 실키 칸만 비움 | 복사하면 끝, 시크릿 0개 |
| **good first issue** | CONTRIBUTING에 위 퀵스타트 + 라벨 | 신입 첫 PR 경로 명시 |

원칙: **실제 Postgres/R2/GitHub OAuth는 배포(Vercel·쿠러그)에서만 필수.** 로컬 기여자가 프론트·콘텐츠만 만질 땐 인프라 세팅 0. (docker조차 부담이면 프론트 전용 mock 데이터 모드도 검토하되, docker Postgres가 기본선.)
