## 1. 개요

### 1.1 프로젝트 배경

T.G.winG는 컴퓨터공학부 학술동아리로, 현재 동아리 내 정보 공유가 Notion 페이지나 개별 채팅에 분산되어 있다. 동아리의 활동과 성과를 외부에 알리고, 내부 멤버 간 지식 공유를 체계화하기 위해 전용 웹사이트를 제작한다.

### 1.2 목적

- 외부적으로는 동아리의 프로젝트와 기술 블로그 아티클을 아카이브하여 동아리 브랜딩을 강화한다.
- 내부적으로는 족보 공유·스터디 관리·인사이트 게시판 등 멤버 전용 기능을 통해 학술 활동의 효율을 높인다.

### 1.3 목표 사용자

| 사용자 | 설명 | 접근 범위 |
|--------|------|-----------|
| 외부 방문자 | 신입 지원자, 타 동아리원, 교수진 등 | Public 페이지만 열람 |
| 일반 멤버 | 현 동아리원 | Public + Private 전체 |
| 운영진 | 동아리 운영진 | Public + Private + 관리자 기능 |

---

## 2. 범위 및 방향성

- 앱이 아닌 **웹**으로 제작하여 접근성을 최우선으로 한다.
- 페이지는 **Public / Private** 두 영역으로 명확히 구분한다.
- 불닭닷컴 About 페이지처럼 **컨셉이 있고 생동감 있는** 분위기를 지향한다.
- winG 동아리만의 **브랜드 아이덴티티(색상, 폰트, 로고)**를 정의하여 전체 페이지에 일관되게 적용한다.
- 첫 페이지에는 강렬한 첫인상을 주는 **스크롤 트리거 기반 애니메이션**을 삽입한다.
- **MVP 핵심 기능을 먼저** 구현하고, 이후 기수에서 기능을 확장하는 방식으로 운영한다.

## 3. 구현 순서

### 1단계 — 기반 세팅 (1~2주)

- 프로젝트 초기 세팅 (Git, 폴더 구조, 코드 컨벤션)
- 디자인 시스템 (색상, 폰트, 공통 컴포넌트)
- GitHub OAuth 로그인 구현
- DB 스키마 설계

### 2단계 — Public 페이지 (2~3주)

- 홈 (랜딩 + 애니메이션)
- 프로젝트 아카이브
- 기술 블로그 아카이브

### 3단계 — Private 핵심 기능 (3~4주)

- 족보 공유
- 인사이트 게시판
- 스터디 / 팀 모집

### 4단계 — 부가 기능 (2~3주)

- 멤버 프로필
- 공지 / 캘린더
- 관리자 페이지 (권한 관리, 멤버 승인)

**전체 MVP 완성 목표: 약 8~12주**

## 4. 기능 요구사항 및 데이터 명세

### 4.1 Public — 홈 (랜딩 페이지)

**기능 설명**

별도의 DB 데이터 없이 정적 콘텐츠로 구성된다. 동아리 소개 문구, 슬로건, 주요 통계(멤버 수, 프로젝트 수, 기수)는 하드코딩하거나 운영진이 관리하는 설정 테이블로 관리한다. 스크롤에 따라 동아리 소개 → 주요 프로젝트 하이라이트 → 기술 블로그 미리보기 섹션이 순차적으로 전개된다.

**저장 데이터**

- 정적 콘텐츠 위주 (별도 테이블 불필요)
- 통계성 데이터는 설정 테이블로 관리 가능

### 4.2 Public — 프로젝트 아카이브

**기능 설명**

동아리원이 진행한 프로젝트를 카드 형태로 나열한다. 디프만(depromeet.com)의 아카이브 구조를 레퍼런스로 삼는다. 기수·기술 스택·수상 여부 기준으로 필터링할 수 있다.

**저장 데이터**

- 프로젝트명
- 설명
- 기수
- 팀원 목록 (User 참조)
- 사용 기술 스택
- GitHub URL
- 배포 URL
- 썸네일 이미지
- 수상 여부 / 수상 내용
- 생성일

### 4.3 Public — 기술 블로그 아카이브

**기능 설명**

멤버들이 작성한 외부 기술 블로그 아티클 링크를 모아 보여주는 페이지다. 링크 등록 시 OG 태그를 파싱하여 썸네일·제목·설명을 자동으로 표시한다. AI·Web·알고리즘·보안 등 분야 태그로 필터링할 수 있다.

**저장 데이터**

- 아티클 제목
- 원본 링크 URL
- OG 썸네일 이미지 URL
- OG 설명
- 작성자 (User 참조)
- 분야 태그
- 등록일, 수정일

### 4.4 Private — 족보 공유

**기능 설명**

파일(PDF, 이미지 등) 형태로만 업로드하며, 글 작성 기능은 제공하지 않는다. 과목명·교수명·학년/학기 태그로 분류하고 검색할 수 있다. 각 파일에 댓글을 남길 수 있어 시험 범위나 출제 경향 팁을 공유할 수 있다. 좋아요·북마크 기능으로 유용한 족보를 상단 노출한다.

**저장 데이터 — 파일**

- 파일명
- 파일 URL (S3)
- 업로드한 멤버 (User 참조)
- 과목명 / 교수명
- 전공·교양 분류 (code)
- 시험 종류 (중간·기말·퀴즈·과제) (code)
- 연도, 학기
- 생성일, 수정일

### 4.5 Private — 스터디 / 팀 모집

**기능 설명**

Notion처럼 파일이 아닌 텍스트 기반으로 모집 공고를 작성한다. 모집 인원·기간·사용 기술 스택·진행 방식을 명시하고, **모집 중·진행 중·완료** 세 가지 상태를 트래킹한다. 완료된 스터디는 결과물과 함께 아카이브로 남긴다.

**저장 데이터 — 스터디 / 공모전**

- 제목 / 본문 내용
- 모집 인원 / 현재 지원 인원
- 시작일 / 종료일
- 상태 (모집 중·진행 중·완료)
- 작성자 (User 참조)
- 생성일 / 수정일

**저장 데이터 — 스터디 참여 멤버**

- 학과, 학번, 이름, 전화번호
- 스터디 (code)

### 4.7 Private — 멤버 프로필

**기능 설명**

기수별 멤버 목록을 제공하며, 각 멤버가 자신의 정보를 직접 관리한다. 활동 기록(작성 글·댓글·스터디 참여 횟수)이 자동으로 집계된다.

**저장 데이터**

- 이름 / 닉네임
- GitHub 아이디
- 프로필 이미지 URL
- 기수
- 역할 (운영진·일반)
- 기술 스택 목록
- 관심 분야
- 개인 포트폴리오 URL
- 자기소개
- 가입일
- 활동 상태 (재학·졸업)

### 4.8 Private — 공지 / 일정

**기능 설명**

운영진이 공지사항을 등록하고, 세미나·OT 등 일정을 캘린더 형태로 제공한다. 멤버는 일정에 참석 신청을 할 수 있다.

**저장 데이터 — 공지사항**

- 제목 / 내용
- 작성자 (운영진, User 참조)
- 생성일 / 수정일

**저장 데이터 — 일정**

- 제목 / 설명
- 시작 일시 / 종료 일시
- 장소
- 종류 (세미나·OT·기타)
- 참석 신청 가능 여부
- 생성일

**저장 데이터 — 참석 신청**

- 일정 ID
- 멤버 ID (User 참조)
- 신청일
